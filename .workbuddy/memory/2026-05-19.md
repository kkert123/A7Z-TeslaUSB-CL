## 哨兵推送BUG修复 (17:31-17:56)

### 问题
用户反馈企业微信推送通知异常：
```
🚨 哨兵事件检测
📍 地点: away
📹 文件: 0 个视频片段
⚠️ 确认码: 054514
```

### 诊断过程
1. SSH到A7Z (paramiko) 检查 SentryClips 目录 → `/mnt/teslacam/TeslaCam/SentryClips`
2. 发现 5/102 个事件文件夹的 event.json 非 JSON 格式，而是Tesla二进制hash

### 修复 (sentry_service.py)
| # | 修改 | 方法 |
|---|------|------|
| 1 | 编码容错 | `_read_event_location`: binary读→检测首字节→多编码回退 (UTF-8→UTF-16→Latin-1) |
| 2 | 空事件过滤 | `_on_new_event`: `file_count<=1` 时跳过推送 |
| 3 | 空事件过滤 | `_on_confirm_request`: `file_count<=1` 时跳过推送 |

## WiFi 页面重复条目修复 (18:18-18:58)

### 问题
手动切换 WiFi 后，/wifi 页面出现多余条目 "WiFi-189-AP"（同一 SSID 显示两条记录）

### 诊断
- NetworkManager 自动创建 profile 名为纯 SSID (如 `189-AP`)
- `wifi_service.py` `switch_wifi()` 硬编码 `con_name = f"WiFi-{ssid}"` (如 `WiFi-189-AP`)
- 同一 SSID 存在两个 NM profile → 页面显示重复
- 共 4 对重复: 189-AP/WiFi-189-AP, HP-00J6O/WiFi-HP-00J6O (另外 2 对 C123/WiFi-C12345, CD/WiFi-H3C_C792C3 实为不同 SSID)

### 修复 (wifi_service.py)
| # | 修改 | 方法 |
|---|------|------|
| 1 | switch_wifi() | 连接前扫描所有 profile 的 SSID，复用已有连接名 (不再硬创 `WiFi-` 前缀) |
| 2 | get_wifi_connections() | 按 SSID 去重，多 profile 同一 SSID 时保留优先级最高的 |
| 3 | NM 清理 | 删除 `WiFi-HP-00J6O` 和 `WiFi-189-AP` 两个冗余 profile |

### 部署
- 备份: `wifi_service.py.bak2`
- 语法验证通过 → `systemctl restart teslausb-web`
- 最终 6 个 connection: HP-00J6O, 189-AP, C123, CD, WiFi-C12345, WiFi-H3C_C792C3 (无重复 SSID)
- API 正常: `/api/wifi/details` 返回 200

## 哨兵预览截图"货不对板"修复 (19:16-20:23)

### 问题
企业微信推送的哨兵预览截图与事件不匹配（画面不是实际触发时刻）

### 诊断
- 事件文件夹名是**结束时间**（14:39:39），event.json 中的是**触发时间**（14:38:38）
- Bug 1: `video_files[0]` 始终选第一个视频段（14:37:28），差 70 秒
- Bug 2: `time_offset = key_ts - folder_name` = -61s（负值）→ 默认 3s
- 结果: 预览显示 14:37:31 的画面，而不是 14:38:38 的关键帧

### 修复 (video_preview.py)
| # | 修改 | 方法 |
|---|------|------|
| 1 | 视频段选择 | 从文件名解析每个视频的时间戳，找到 key_ts 所在的段 |
| 2 | 时间偏移 | `time_offset = key_ts - 所选视频时间戳` (如 9s) |
| 3 | 正则修复 | 用 `vf.name` 而非 `str(vf)` 避免匹配到文件夹名 |

### 测试验证
- 事件 14-39-39: front/back/left/right 全部提取在 offset=9.0s (14:38:38, 正确!)
- 四宫格预览生成成功: 143KB, 74KB thumb
- 对比修复前: offset=3.0s (错误画面, 差 67 秒)
